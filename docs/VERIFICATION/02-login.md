# Login — Verification

**Figma reference:** node 49:268 (`.design-reference/screens/login-mobile-no-entry__49-268`)
**Implementation:** src/features/auth/LoginScreen.tsx
**App screenshot:** /tmp/bonaca-audit/02-login.png

## Verdict
Pass with minor issues — layout, copy, colors and component structure closely track Figma; the deviations found are sub-pixel color drift, a non-bottom-anchored CTA-block layout strategy that differs structurally from Figma's fixed-gap spec, and missing interactivity/accessibility on the country-code control.

## Code quality / responsiveness issues
| # | Issue | File:line | Severity | Notes |
|---|---|---|---|---|
| 1 | Country-code control is not interactive | src/features/auth/components/MobileNumberField.tsx:14-18 | medium | The `+91 ⌄` block is a plain `View` with a `Text` and `IconChevronDown` — there is no `Pressable`/`TouchableOpacity`, `onPress`, or any handler. The chevron-down icon strongly implies a tappable country-code selector (the screen is literally named "Login - Mobile No. Entry" with India as the only seeded code), but tapping it does nothing. This isn't a Figma-vs-code visual mismatch (Figma is a static mock with no prototype interaction defined either), but functionally the control is dead UI that looks actionable. |
| 2 | No accessibility props on the mobile number `TextInput` | src/features/auth/components/MobileNumberField.tsx:20-28 | medium | The `TextInput` has no `accessibilityLabel`/`accessibilityHint`. A screen-reader user gets only the placeholder ("Mobile number") read out via default behavior, with no indication that a `+91` prefix is implicitly part of the value, and no `accessibilityRole`. Compare to `PrimaryButton.tsx:17-18` and the Privacy Policy `Pressable` in `LoginScreen.tsx:41`, which both correctly set `accessibilityRole`/`accessibilityLabel` — this component is the outlier. |
| 3 | Country-code chevron icon and divider have no accessibility treatment | src/features/auth/components/MobileNumberField.tsx:17,19 | low | Since the whole control isn't a `Pressable` (see issue 1), there's also no `accessibilityRole="button"` / label (e.g. "Country code, +91") that a future tappable version would need — flagging alongside issue 1 since fixing one implies fixing the other. |
| 4 | Fixed-pixel hero height not screen-relative | src/features/auth/LoginScreen.tsx:24 (`height={464}`) | low | `464` is a literal hardcoded pixel value (matches Figma's exact 464px hero height at the 390×844 reference frame), passed as a prop to `AuthHero`. On shorter devices (e.g. iPhone SE, 375×667) a fixed 464px hero leaves only ~203px for the white card before any keyboard appears, likely forcing the field/CTA off-screen or very cramped; on taller devices (e.g. iPhone 16 Pro Max, 430×932) it under-fills the proportional header height Figma's design implies. Not wrong at the reference size, but not safe-area/screen-relative either. |
| 5 | `KeyboardAvoidingView` has no `keyboardVerticalOffset` and untested behavior on smaller devices | src/features/auth/LoginScreen.tsx:19-21 | medium | `behavior="padding"` is set for iOS only (`undefined` on Android, which needs `behavior="height"` or a windowSoftInputMode config to avoid the input being covered — RN's Android default of doing nothing here can leave the keyboard overlapping the input on smaller Android screens). Given the hero is a fixed 464px tall (issue 4) and the field sits below it, on a short device the available space for `card` content above the keyboard is tight; no `keyboardVerticalOffset` is supplied to compensate for the status bar/notch height, which `KeyboardAvoidingView` docs recommend setting explicitly when a non-full-screen header precedes the content. |
| 6 | Disabled button state uses a sensible non-hardcoded value | src/features/auth/components/PrimaryButton.tsx:32-34 (`buttonDisabled: { opacity: 0.5 }`) | none (informational) | Per task instructions, a disabled-state visual difference is expected and not itself a Figma deviation. Confirmed the disabled treatment is `opacity: 0.5` applied to the existing `Colors.accent` background — not a separate hardcoded gray/hex — so this is a clean, token-respecting implementation choice, not a stray value. |

## Confirmed correct
- Hero/header gradient stop colors: exact hex match (`#090c2c` → `#555ec2`) between Figma fill stops and `Colors.headerGradientStart`/`headerGradientEnd`.
- Hero height: Figma's gradient header is exactly 464px tall at the 390×844 reference frame; implementation passes `height={464}` to `AuthHero` — an exact match (see issue 4 above for the responsiveness caveat, which is separate from correctness-at-reference-size).
- White card top corner radius: Figma's card (node 49:289) uses `cornerRadii: [48,48,0,0]`; implementation uses `Radii.cardTop = 48` for `borderTopLeftRadius`/`borderTopRightRadius` (LoginScreen.tsx:58-59) — exact match, including the card overlapping the gradient via `marginTop: -48` to recreate Figma's card sitting flush against/over the header.
- "Login or Signup" title: text, font (DM Sans, weight 600, size 20, lineHeight 28, center-aligned), and color (`#1f2d2b` = `Colors.textPrimary`) all match Figma node 49:291 exactly.
- Subtitle "Enter your mobile number to get started": text, font (DM Sans, weight 400, size 14, lineHeight 20), and color (`#4e535a` = `Colors.textSecondary`) match Figma node 49:302 exactly.
- Mobile number field: height 48, border radius 8 (`Radii.cta`), white background, all match Figma node 49:293/49:294 exactly.
- "+91" country code text: DM Sans weight 400, size 16, lineHeight 20, color `#1f2d2b` — matches Figma node 49:295 exactly.
- Chevron icon: `IconChevronDown` size 20, color `Colors.textPrimary` — matches Figma's 20×20 arrow_drop_down group (node 49:296) in size and dark color treatment.
- Divider line color: Figma stroke `#e3e5e5`(~`rgb(227,229,229)`) vs implementation `Colors.inputBorder = '#e3e4e6'` — same low-severity 1-unit drift as discrepancy #1, otherwise correctly implemented as a 1px vertical divider.
- "Mobile number" placeholder text: DM Sans weight 400, size 16, lineHeight 20 — matches Figma node 49:300 (color noted as low-severity drift in discrepancy #2).
- Send OTP CTA: width spans full card content width (358px in Figma = 390 frame − 16px margins each side; implementation achieves this via `paddingHorizontal: 16` on the card + full-width button), height 56, border radius 12 (`Radii.button`), background `#575fb4` (`Colors.accent`) — exact match to Figma node 49:303.
- "Send OTP" label: DM Sans weight 600, size 18, lineHeight 24, white text, center-aligned — matches Figma node 49:307 exactly.
- Privacy Policy link: text, DM Sans weight 400, size 14, lineHeight 20, color `#1f2d2b` (`Colors.textPrimary`), underlined — matches Figma node 49:309 exactly, including the underline treatment implied by the link styling.
- Gap between CTA button and Privacy Policy link: Figma has exactly 24px (node 49:313 `itemSpacing: 24`); implementation's `ctaBlock` style uses `gap: 24` — exact match.
- Colors are sourced from `src/theme/tokens.ts` constants throughout (`Colors.accent`, `Colors.textPrimary`, `Colors.textSecondary`, `Colors.white`, `Colors.inputBorder`, `Colors.placeholderText`, `Colors.headerGradientStart/End`, `Radii.cardTop/cta/button`) rather than hardcoded hex/number literals scattered through the components — good adherence to CLAUDE.md's token convention.
- `disabled` state on the Send OTP button correctly gates on `mobileNumber.length === 10` (LoginScreen.tsx:16), a sensible 10-digit Indian mobile number validation matching the `+91` country code context, and `maxLength={10}` on the `TextInput` (MobileNumberField.tsx:27) prevents over-typing.
- `useSafeAreaInsets` is used to pad the CTA block's bottom (`insets.bottom + 16`, LoginScreen.tsx:35), correctly avoiding the home indicator / safe-area on notched devices rather than a hardcoded bottom value.
- `KeyboardAvoidingView` is present and platform-gated correctly for the `behavior` prop value (`'padding'` on iOS) — base keyboard-avoidance exists, even if not fully tuned (see code-quality issue 5).
